Krzysztof Manuszewski ABSTRACT Progressive Web Applications (PWA) are single-page applications that behave like traditional desk-top or mobile applications. Although typically developed using JavaScript-based frameworks, Blazor presents an alternative approach using .NET technology. Blazor allows developers to build interactive web applications using C# instead of JavaScript, potentially offering new advantages for PWA develop-ment. However, the specific benefits and limitations of using Blazor for PWAs have not been explored. This study aims to fill this gap by evaluating two functionally identical applications, one developed using Blazor and the other with Vue.js, a widely adopted JavaScript framework. The development pro-cess for both applications is described in detail, highlighting the architectural decisions, implementation challenges, and specific techniques used to meet the requirements of PWA. The applications were compared by initial load time, runtime performance, and application size. In addition, the developer experience was measured by examining the volume of code and the startup time of the development server. The findings show that Blazor with WebAssembly-based client-side rendering is a suitable option for developing PWAs, especially when performance is not the primary concern. It meets essential PWA requirements such as offline support, installability, and background synchronization. It also provides advantages in development efficiency and maintainability by reducing the volume of code needed for application logic. However, its larger application size and performance trade-offs make it less suitable for performance-critical applications compared to frameworks like Vue.js. INTRODUCTION 1.1. Context of the study Over the last decade, we have seen significant changes in the field of web development, driven by advances in web frameworks, increased computer performance and bandwidth, and shifting user expectations. One of them started in 2015, when two Google employees, developer Alex Russell and designer Frances Berriman, introduced the concept of Progressive Web Applications (PWAs). They saw PWAs as a way to bring the best user experiences from native mobile apps to the web by using features such as service workers and web app manifests. [1] The usage and popularity of the new web application standard have increased significantly since it was introduced. Flipkart, India’s largest e-commerce platform, was one of the first to use this technol-ogy in their application, Flipkart Lite. This new application dramatically improved the user experience compared to the previous mobile platform. The adoption of the new standard resulted in users spend-ing three times more time on the website, a 40% higher reengagement rate, a 70% higher conversion rate among users who reached the site using the "Add to Homescreen" function, and a three times lower data usage.[2] Other major companies have recognized the value of PWAs and have joined the trend. Platforms such as Twitter, Forbes, Alibaba, and many others benefited significantly, with lower data consumption, significant increases in conversions, monthly active users, as well as faster loading times and higher user sessions and engagement. [1] The advantages of PWAs are not limited to large corporations, as they can offer significant benefits to startups by accelerating product development and reducing costs. Unlike traditional native apps, they can be developed, tested, and deployed quickly, allowing startups to iterate through ideas cost-effectively. This makes them a powerful tool for companies that want to accelerate the development of prototypes and effectively match product to market. Using PWAs, startups can allocate resources more efficiently to innovation and refinement of their offerings, rather than investing heavily in native app development. [3] Given the numerous benefits provided by PWA, the technology has been widely adopted since its introduction in 2015. However, when deciding to adopt this web standard, developers face a crucial question: which framework and programming language to use? Traditionally, JavaScript has been the main language for web development, and frameworks such as Angular, React, and Vue.js were designed to facilitate PWA creation. However, the web development environment is changing and new technologies are emerging that can provide attractive alternatives. This thesis examines Blazor technology as one of the alternatives to JavaScript frameworks in PWA development. Blazor, a technology developed by Microsoft, is a modern front-end web framework that allows developers to create interactive web applications using mainly C# and .NET, while still allowing for integration with JavaScript. It supports two hosting models: server-side and WebAssembly. The server-side model performs application logic on the server and handles the user interface via a real-time SignalR connection, whereas Blazor WebAssembly runs code client-side in the browser on a WebAssembly-based .NET runtime. [4] 1.2. Goals of the Study The purpose of this study is to investigate the suitability of using Blazor technology to implement progressive web applications. The study covers the implementation of selected aspects of PWA in Blazor and compares them with a reference technology built on JavaScript, in particular Vue.js. To achieve this goal, the study answers the following research questions: RQ1 What are the technical capabilities of Blazor in meeting PWA requirements? RQ2 What are the advantages and disadvantages of using Blazor to build PWAs in comparison to JavaScript-based technologies? 1.3. PROGRESSIVE WEB APPLICATION The universal accessibility, strong security, and open standards of the Web make it an ideal platform to deliver software that anyone can access and share. Web applications work on any device with a single codebase and are easy to deploy without the need for packaging, content reviews, or dealing with update delays. Users are guaranteed to get the most recent version every time they access an application.[9] In contrast, platform-specific applications offer deep integration with devices, enabling rich func-tionality and offline use. However, they are often tied to specific platforms, making cross-platform compatibility challenging. Progressive Web Applications merge the strengths of both, combining the accessibility of web applications with the user experience similar to native applications.[9] Progressive Web Applications (PWAs) are called "progressive" because they rely on a technique known as progressive enhancement. This approach focuses on developing a core experience that works across all devices and browsers, while allowing for more advanced functionality on more capable devices. With progressive enhancement, every user has access to the application’s core functionality, regardless of device limitations. However, as the device and browser engine’s capabilities increase, the application will be able to unlock new features and provide better performance. On more advanced devices, PWA may offer offline access, push notifications, or integration with hardware components such as the camera or GPS. [10] Progressive enhancement is an important characteristic of Progressive Web Applications, but there are more factors that contribute to delivering a native app-like experience. Google identified five es-sential criteria that also play a significant role in making a web application installable, reliable, and user-friendly on different devices and browsers.[11] Key aspects include: 3.1. Service Workers Service workers are a technology that brings multiple capabilities to web applications. The most significant of these is the ability to operate in scenarios where Internet connectivity is either limited or unavailable. They function as proxy servers, situated between the web application, the browser, and the network. This enables the app to perform tasks such as offline access, interception of network requests, caching of resources, and providing fallback content when the network is not accessible. [13] A service worker is an event-driven background task, created via a script, that operates within a specific origin (the combination of protocol, domain, and port) and a defined path, which determines the scope of its control. It has the form of a JavaScript file that manages the associated web page by intercepting and modifying navigation and resource requests, and caching resources. When an app requests a resource, it can serve it from the cache, fetch it from the network, or generate it locally, providing a platform-like experience even when offline. Running in a separate worker context, service workers are non-blocking, fully asynchronous, and operate on a different thread from the main app, ensuring they don’t interfere with primary processes.[13] [14] 3.3.1. Lifecycle of service workers In order to successfully implement service workers in PWAs, it is essential to gain an understanding of their lifecycle. This involves a number of distinct phases, which are illustrated in the Figure 3.1. Registration The journey of a service worker begins with registration. When a user first opens the website, all network requests initially go directly to the server because the service worker hasn’t taken control yet. The registration process starts by checking if the browser supports the Service Worker API. Once this is confirmed, PWA registers the service worker script. The location of the script determines its scope; placing it at the root gives it control over the entire domain, while placing it in a subdirectory limits its scope. [14] Installation Once it is registered successfully, the installation phase begins. This process happens silently in the background, without needing user permission, even if the user does not install PWA.If the installation fails, the service worker does not take control of the page. The successful installation fires the install event, which occurs only once per service worker. Developers can listen for that event to prepare the service worker for use, typically by creating a cache and storing static assets needed for offline functionality.[15] Activation After successful registration and installation, the service worker state changes to ’activating’. During the activation, tasks such as cleaning up old caches can be performed, which is important for service worker updates. For a new service worker, activation occurs immediately after installation. Once activation is complete, the state changes to ’activated’. By default, a new service worker will only take control of the pages on the next navigation or page refresh. This ensures that any currently active pages continue to function under the control of the previous service worker until they are reloaded. [16] Termination or Action Service workers are designed to be resource efficient, which means that they run only when neces-sary. After performing their tasks, service workers enter an idle state. If there are no active clients within their scope, the service worker will be terminated by the browser after being idle for a few seconds or working too long, to conserve memory and Central Processing Unit (CPU) resources. However, ser-vice workers are automatically restarted when needed, such as when a new network request occurs or a message is sent to them. This ensures that service workers are available to perform necessary actions without consuming resources unnecessarily.[17] 3.3.2. Caching and offline functionality Caching is a fundamental aspect of developing PWAs that enables them to be both fast and reliable and less dependent on network conditions. Service workers allow developers to cache specific assets independently of the main application thread, providing greater control over how and when resources are stored on users’ devices. However, this also brings with it the responsibility to use storage space and bandwidth wisely, as these assets are downloaded and stored locally. The key is to achieve an optimal balance between caching enough assets to ensure a fast, reliable, or offline experience without overloading the user’s device with excessive data. [18] To achieve fast load times, it is essential to cache the components required to render the user interface. This typically includes the main page’s HTML, CSS stylesheets, images, JavaScript files, web fonts, etc. By caching these assets, PWA is able to rapidly display a fully functional interface. This process typically occurs during the Service Worker’s installation. [18] In addition to these assets, providing an offline experience requires caching at least an offline page. This page informs users that they are offline and that certain requests cannot be completed at the moment. BLAZOR Blazor is an open-source web framework developed by Microsoft as part of the .NET platform. It has been designed to enable the creation of interactive web applications using C# and .NET Core as an alternative to JavaScript for front-end development. Blazor applications are constructed with components, which are fundamental UI elements written as .NET C# classes that define dynamic UI rendering logic and manage user events. Typically, they are implemented using a Razor syntax that combines C# and HTML markup in files with the .razor extension. [24] Blazor is operating with two rendering modes: Client-Side Rendering (CSR) and interactive Server-Side Rendering (SSR). In SSR, user interactions and UI updates are handled through a real-time connection to the browser, established via SignalR. This technology employs WebSockets to transmit events and UI changes between the client and the server. In CSR, WebAssembly is used to run C# code directly in the user’s browser, allowing each user to have their own local instance of the app. In contrast, Blazor Server operates with a single instance of the application on the server, to which multiple users connect simultaneously.[25] With the release of .NET 8 in 2023, Blazor introduced interactive Auto (automatic) rendering, which enables the use of both server-side and client-side rendering within a single web application. Initially, the server-side ASP.NET Core runtime handles content rendering and interactivity. Once the Blazor bundle is downloaded and the WebAssembly runtime is activated on the client, subsequent rendering and interactivity shift to the client side. This approach typically offers the fastest app startup experi-ence.[26] However, creating Progressive Web Applications with Blazor is only feasible by developing ap-plications that run entirely on WebAssembly in the browser, without involving a server. Server-side rendering requires a stable network connection to maintain an active session between the user and the server, which conflicts with the offline functionality of PWAs. The standalone Blazor WebAssembly apps deploy assets as static files to a web server, and once downloaded, they can be cached and run offline as PWAs.[24] 4.1. WebAssembly WebAssembly is a type of code designed to run in modern web browsers, offering significant per-formance enhancements. In contrast to JavaScript, which is the dominant language in web develop-ment, WebAssembly is not typically created manually. Instead, it functions as a compilation target for languages such as C++ or C# and others. This allows developers to execute code written in these lan-guages on the web at near-native speeds, enabling the creation of high-performance web applications that were previously impractical due to the limitations of JavaScript. [27] As a high-level language, JavaScript can be resource-intensive to download, parse, and compile, which may lead to challenges in delivering a smooth user experience on devices with limited comput-ing power, especially for more demanding use cases. WebAssembly uses binary format files which are smaller than equivalent files containing JavaScript code, reducing the time required for network downloads. Additionally, because the WebAssembly code is precompiled, it can be executed directly by a browser. These features lead to improved performance, enabling faster loading and execution. As a result, WebAssembly enables the development of advanced web applications such as 3D games, virtual and augmented reality, complex image and video processing, and more.[27] WebAssembly is platform independent, which means it will run the same code on any device with a compatible browser, regardless of operating system or hardware architecture. It runs in a secure, sandboxed environment separate from the host machine. This reduces the possibility of malicious code interacting with the system outside its intended boundaries.[27] 4.2. Blazor WebAssembly 4.2.1. Project Startup To understand how the applications written in C#, after compilation into WebAssembly, are fully ex-ecuted in a browser, it is important to know the initial process behind starting the Blazor WebAssembly application. The startup process begins with the initial page load. The browser sends a GET request to the server to fetch the main page of the Blazor WebAssembly application, which is typically a static in-dex.html file. The browser then parses HTML and loads linked resources such as CSS, JavaScript, and images. During this process, the browser initially renders the static HTML, displaying a place-holder, where the Blazor application will later render its content.[28] As the Blazor WebAssembly initialization begins, the blazor.webassembly.js script is loaded and executed. This JavaScript file handles the loading and initialization of the Blazor application within the browser. First, it downloads the configuration file, blazor.boot.json , which contains a list of all required framework and application files. These include .NET assemblies and the dotnet.wasm file, which is a complete .NET runtime compiled into WebAssembly binary format. After downloading all the listed resources, the WebAssembly .NET runtime is initialized, loading the Blazor framework and the application. At this point, the application runs entirely in the client’s browser. [28] 4.2.2. User interface updates The role of the blazor.webassembly.js script does not end with the loading and initialization of the Blazor application. This part of the framework is crucial for performing the necessary DOM updates and providing APIs for scenarios requiring JavaScript interoperability. The use of JavaScript in these scenarios is necessary due to WebAssembly’s limitations, as it cannot directly manipulate the DOM or interact with Web APIs. Although future advancements in WebAssembly may reduce the dependency on this file, at this moment it is essential for UI changes. [25] When a user initiates a UI update, such as navigating from one page to another, the event is in-tercepted by Blazor’s JavaScript runtime (blazor.webassembly.js), which communicates with the Blazor framework running in the WebAssembly runtime (dotnet.wasm). The Blazor router processes the nav-igation request by identifying the corresponding component, creating an instance of it, and executing its lifecycle methods. [25] During this process, Blazor constructs a Render Tree, which serves as an internal representation of the UI structure of the component. To optimize updates, Blazor employs an incremental DOM approach. Instead of re-rendering the entire UI, it compares the newly generated Render Tree with the previous one to determine the minimal set of required changes. This comparison results in a differential render tree that identifies elements that need to be added, updated, or removed. The calculated updates are then sent back to the JavaScript runtime, which applies them to the actual DOM. [29] 5. VUE 5.1. Overview Vue.js was selected as the reference technology to examine how Blazor differs from a widely used JavaScript framework in the development, performance, and capabilities of Progressive Web Applica-tions. Released by software developer Evan You in 2014, Vue.js is an open-source framework that has gained significant popularity. Built on standard HTML, CSS, and JavaScript, Vue.js offers a declarative, component-based programming model that enables the efficient development of user interfaces of any complexity.[30] Vue.js uses a template syntax that extends standard HTML that declaratively describes the HTML output based on the application’s JavaScript state. The framework’s reactivity system automatically tracks changes in the state and efficiently updates the Document Object Model when modifications occur. Similarly to Blazor’s use of .razor files, Vue.js encapsulates each component within a file with .vue extension, which contains the logic, template, and styles in a single unit.[30] Components can be written in two different API styles: Options API and Composition API. The Options API, which was the original way to define components, organizes component code by options such as data, methods, and computed properties. The Composition API, introduced in Vue 3, pro-vides an alternative way of defining component logic using functions, which can be more suitable for organizing complex logic and promoting code reuse.[30] Vue.js, like other popular JavaScript frameworks, supports TypeScript. TypeScript is a strongly typed programming language that builds on JavaScript and helps detect common errors through static analysis at build time, reducing the likelihood of runtime errors in production. It also makes refactoring code in larger applications more reliable and enhances the developer experience with type-based auto completion in Integrated Development Environment (IDE).[31] 5.2. Rendering mechanism Vue.js rendering mechanism utilizes the Virtual DOM concept, which serves as an in-memory repre-sentation of the UI. Instead of inefficiently manipulating the real DOM directly, Vue.js creates lightweight JavaScript objects called virtual nodes that mirror the structure of actual DOM elements. When the ap-plication’s state changes, Vue.js re-renders the Virtual DOM to produce a new virtual tree. Then it efficiently compares this new tree with the previous one to identify the minimal set of changes needed. This process allows Vue.js to update only the parts of the real DOM that have changed, significantly optimizing performance.[32] 5.3. Build tools Vue.js currently uses Vite as its default build tool. Vite is a modern build tool developed by the Vue team, designed to provide a faster and more efficient development experience, offering an op-timized workflow for both development and production builds. Unlike traditional bundlers, Vite takes advantage of native ECMAScript Modules (ESM), making it possible to deliver an optimized workflow for development builds.[33] Vite improves the development phase by enhancing dev server start times and optimizing module handling. It separates modules into dependencies and source code: dependencies are pre-bundled using the fast "esbuild" tool since they change infrequently, accelerating initial load times. The source code is served over native ESM, allowing the browser to handle module loading and transform code on demand, reducing unnecessary processing. Vite enhances Hot Module Replacement by precisely updating only the changed modules, keeping updates fast regardless of application size and preserving application state without full page reloads. It also utilizes browser caching through HTTP headers to minimize server requests during reloads. By offloading tasks to the browser and minimizing processing, Vite maintains high performance even as the application grows, resulting in a faster and more efficient development experience.[33] For the production build, bundling is still used because unbundled ESM can be inefficient due to extra network round trips from nested imports, even with HTTP/2. Bundling optimizes performance through tree-shaking, lazy-loading, and chunk splitting for better caching. Vite uses Rollup 1 for this process because it offers a more flexible plugin API, which fits Vite’s infrastructure better than esbuild, despite esbuild’s faster performance.[33] 1 is a module bundler for JavaScript 6. For 5,000 players, Vue’s sort time is actually higher than Blazor’s, indicating WebAssembly’s advantage in computational tasks that do not involve DOM manipulation or framework-specific operations. The most significant performance difference is observed in the UI diffing phase, where Vue com-pletes the process significantly faster than Blazor. Vue’s diffing algorithm processes changes 30 times faster for 100 players, and this difference increases to 38 times faster for 5000 players, where the UI diffing phase takes nearly three seconds in Blazor. Both frameworks perform UI diffing by com-paring the new UI representation with the previous state, but their approaches differ fundamentally. Blazor constructs a Render Tree and executes its diffing algorithm within the .NET runtime running on WebAssembly, which then requires communication of DOM updates across the WebAssembly-JavaScript boundary. In contrast, Vue employs a Virtual DOM diffing approach that operates entirely within JavaScript, benefiting from modern JavaScript engine optimizations and avoiding the additional overhead caused by boundary-crossing operations. This difference in execution environment con-tributes to the performance gap observed between the two frameworks. Both frameworks appear to compute the minimal set of UI changes with similar accuracy, as evi-denced by the comparable rendering times observed in the browser rendering phase. The data show that rendering times remain relatively consistent between Vue and Blazor across all dataset sizes, indi-cating that both frameworks are generating similar sets of DOM operations for the browser to process. 7.4. CONCLUSION This study examined the suitability of using Blazor technology for the implementation of Progres-sive Web Applications and compared it with a JavaScript-based framework, Vue.js. To achieve this, the research focused on two key questions: the technical capabilities of Blazor in meeting PWA re-quirements and the advantages and disadvantages of using Blazor in comparison to JavaScript-based technologies. The findings of this study indicate that Blazor technology enables the development of Progres-sive Web Applications through its WebAssembly-based implementation, which runs entirely within the browser. Essential PWA features including installability, offline access, and background synchroniza-tion, were successfully implemented in both applications developed during this research. Although Blazor primarily supports C# development, it provides interfaces to execute JavaScript code when necessary, such as requesting notification permissions. While Blazor includes a default PWA template with basic service worker implementation for offline access, this research evaluated Blazor’s compatibility with the widely adopted Workbox library, which simplifies service worker development. The integration process revealed differences between frame-works: Vue.js offers a dedicated plugin with native Workbox compatibility, enabling automatic service worker generation, whereas Blazor requires manual integration by importing Workbox from a Content Delivery Network and utilizing Workbox CLI for precaching implementation. Despite these procedural differences, the service worker was successfully implemented within the Blazor framework. Beyond the implementation of the service worker, other aspects of Blazor must be evaluated when considering its suitability for PWA development. The most significant challenge of using this technology is its performance. Blazor WebAssembly applications require downloading the .NET runtime and nec-essary dependencies before execution, resulting in a larger application payload. This leads to longer startup times compared to JavaScript frameworks such as Vue.js, which are optimized for fast initial rendering. The delay is especially noticeable on lower-end devices or under poor network conditions, where extended loading times can negatively impact the user experience. In addition to load-time concerns, Blazor WebAssembly presents further runtime performance con-siderations. Although it enables the execution of C# code directly in the browser, WebAssembly can-not directly manipulate the DOM. Instead, UI changes must be captured by JavaScript, passed to WebAssembly where the C# code is executed, and then returned to JavaScript. This architectural approach introduces overhead compared to Vue’s direct DOM manipulation. However, in pure compu-tational tasks, such as sorting datasets analyzed in this research, Blazor gains an advantage as the size of the dataset increases. The difference in performance is also noticeable during the development process. Blazor’s server startup is more time consuming than Vue’s, due to the need to initialize the .NET runtime, compile Ra-zor components, and configure dependency injection. In contrast, Vue benefits from Vite’s lightweight, module-based compilation process, enabling faster startup and a more responsive development work-flow. The main advantage of using Blazor for PWA development is its ability to significantly reduce overall code volume, resulting in a simpler and more maintainable codebase. Using C# for application logic allows developers to write more concise code. The built-in dependency injection framework simplifies service configuration by eliminating redundancy, and the convention-based routing system automati-cally generates routes from the project structure, eliminating the need for separate routing configuration files. In the application developed for this research, these features reduced the number of code lines by 20–30% compared to Vue. Based on the findings in this study, Blazor WebAssembly is a suitable option for PWA development in scenarios where performance is not the primary concern. Blazor fully meets all PWA requirements, providing features such as offline support, background sync, and installability. Although its larger ap-plication size and DOM updates across the WebAssembly-JavaScript boundary impact performance sensitive applications, it offers significant advantages in development efficiency and maintainability. It is especially suitable for teams already experienced with .NET, allowing developers to work within a familiar environment and reducing the learning curve. Using C# for front-end applications simplifies code maintenance and ensures better long-term support, making it practical for in-house applications where rapid development and ease of maintenance take priority over performance requirements. Future studies could explore several important areas to deepen the understanding of Blazor’s po-tential in PWA development. Although this study compared the volume of code between Blazor and a Vue.js for a relatively simple application, examining larger and more complex projects could offer greater insight into how each technology manages scalability, maintainability, and project structure. Another valuable direction for future research involves the security aspects of Blazor PWAs, with a particular focus on user authentication and authorization. This could include evaluating Blazor’s sup-port for modern protocols such as OAuth and OpenID Connect, and comparing these implementations with those used in JavaScript-based PWAs. Additionally, it would be beneficial to assess how Blazor handles secure token management, permission scoping, and session persistence, especially in offline scenarios. Furthermore, with the release of .NET 8, Blazor introduced an interactive auto rendering mode designed to improve initial load times by starting with server-side rendering and transitioning to We-bAssembly once the necessary client-side resources are loaded. Although this approach currently lacks support for PWA development, future enhancements may introduce such capabilities. If PWA support is introduced in the future, it would provide an important area for research, especially to under-stand its impact on startup performance and how mixed rendering modes work with key PWA features.
